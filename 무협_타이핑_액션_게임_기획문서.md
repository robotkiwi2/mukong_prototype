# 무협 타이핑 액션 게임 — 기획문서

> 초안 / 컨셉 단계
> 작성 기준일: 2026-06-07

---

## 1. 개요

**한 줄 컨셉**
이동·방어·공격 등 모든 행위를 **타이핑(+엔터)으로 시전**하는, 웹 기반 실시간 무협 액션 게임.

| 항목 | 내용 |
|------|------|
| 장르 | 실시간 액션 / 무협 |
| 플랫폼 | 웹 (HTML/JS) |
| 입력 | 키보드 타이핑 + 엔터 |
| 핵심 판타지 | 구결(口訣)을 외워 무공을 펼치는 무림 고수 |

이 게임에서 타이핑은 단순 조작이 아니라 **세계관 내 행위 그 자체**다. 무공에 구결과 초식 명칭이 존재하는 무협 세계관과 "어구를 입력해 무공을 발현한다"는 메커닉이 자연스럽게 맞물린다. 따라서 무공을 배운다는 것은 곧 **해당 어구를 익히는 것**이며, 숙련되면 줄임말·단축어로 더 빠르고 능숙하게 시전하게 된다.

---

## 2. 디자인 기둥 (Design Pillars)

1. **입력이 곧 시전이다.** 타이핑하는 행위와 무공이 펼쳐지는 행위가 일대일로 대응한다.
2. **어구의 길이가 밸런싱의 화폐다.** 강한 무공일수록 길게 쳐야 한다 — 이것은 다른 장르엔 없는, 타이핑 게임 고유의 밸런싱 수단이다.
3. **직렬 입력의 한계를 무공으로 푼다.** 손은 한 번에 한 문장만 친다. 다수의 위협은 더 빠른 타이핑이 아니라 광역 무공으로 해결한다.
4. **무공은 기본행위의 조합이다.** 작은 행위들을 엮은 것이 초식이며, 학습은 더 긴 조합을 한 단어처럼 쓰는 청크(chunk)의 획득이다.
5. **지식과 실력은 다른 축이다.** 어떤 초식을 아는가(무학)와, 기본행위를 얼마나 단련했는가(내공·숙련)는 분리된 성장 축이다.

## 3. 핵심 게임플레이 루프

전장에는 여러 캐릭터가 존재하고, 적의 공격(장풍 등)이 공간을 가로질러 날아오는 것이 보인다. 플레이어는 그것을 보며 회피하거나 방어 초식을 시전한다. **공격의 비행 시간이 곧 대응에 쓸 수 있는 시간**이며, 이를 놓치면 피격된다.

```
적의 공격 인지 → 상황 판단(단일/포위) → 적절한 어구 입력
→ 정확도·타이밍 판정 → 효과 발현(회피·방어·반격) → 다음 합
```

여기서 게임의 실력은 단순 손속도가 아니라, **압박 속에서 상황에 맞는 초식을 떠올려 정확히 꺼내는 판단 + 실행**이다.

## 4. 전투 시스템

### 4.1 실시간 공간 전투

공간과 위치가 존재하는 실시간 액션이되, 모든 행위가 타이핑으로 이뤄진다. 적 공격의 **비행 속도가 곧 난이도 조절 레버**다. 멀리서 느리게 오면 정식 초식명을 다 칠 여유가 있고, 코앞에서 빠르게 오면 짧은 단축어가 아니면 막을 수 없다. 따라서 타이핑 숙련이 곧 생존 능력으로 직결된다.

### 4.2 입력 = 시전, 정확도 등급제 (graded)

오타가 나도 무공은 **시전되되 효과가 감소**한다. 전부-아니면-전무(binary)의 실패가 아니라 리듬게임의 Perfect/Great/Good처럼 항상 무언가는 일어나, 플레이어가 흐름에서 튕겨나가지 않는다. 이는 한글 IME·조합형 오타 문제도 완화한다.

**단, 정확도가 반드시 추구할 가치가 있도록** 다음 장치를 둔다 (정확도가 무의미하면 "대충 빨리 막 치기"가 최적 전략이 되어버린다):

- **막기 연동:** 정확도가 임계치 아래인 엉성한 초식은 상대가 자동으로 파해(破解)·흘려보내 효과가 0에 가까워진다.
- **비선형 보상:** 95% 정확도가 70%의 단순 비례 이상으로 압도적인 효과를 내도록 곡선을 가파르게.
- **극단적 오타 = 주화입마:** 심하게 틀리면 기(氣)가 역류해 자기 내력이 깎인다. "막 치기"가 비효율을 넘어 위험해진다.

### 4.3 직렬 입력 vs 병렬 위협의 해소

손은 한 번에 한 문장만 친다(직렬). 그러나 실시간 다중 위협은 병렬이다. 이 모순은 **광역 무공**으로 푼다 — 한 번의 시전으로 효과를 사방에 퍼뜨린다. 광역 무공은 역할별 삼각 구도를 이룬다.

| 종류 | 예시 | 역할 |
|------|------|------|
| 버티기(무효화) | 금강불괴 천하무골 | 제자리에서 주변 공격 무효화 |
| 빠지기(회피·이동) | 제운종 신출귀몰 | 주변 적을 한 번에 회피·이탈 |
| 받아치기(광역 반격) | (예: 검기 폭발) | 사방의 적을 동시 타격 |

광역 무공은 **어구가 길다.** 따라서 짧은 반응 윈도우 안에는 칠 수 없고, 적이 모여드는 것을 미리 읽고 **선제적으로 시전**해야만 완성된다. 입력 길이 자체가 "선견지명을 요구하는" 밸런싱 장치로 작동한다.

보조 장치: 적의 공격을 시간차로 엇갈리게 배치(거의 항상 대응할 위협이 하나만 보이도록), 시전 시작 시 시간 둔화(심상의 세계 — 직렬 입력이 병렬 위협을 따라잡을 여유) 등.

### 4.4 시전 중 끊기(파해)와 콤보

긴 무공을 시전하는 동안 플레이어는 무방비이며, 빠른 적이 그 틈을 파고들면 시전이 **끊긴다**("초식이 깨졌다"). 이는 광역기가 공짜 승리가 되지 않게 하는 카운터플레이다. 여기서 자연스럽게 콤보 플레이가 발생한다: 짧은 견제기로 적을 잠깐 굳혀놓고, 그 틈에 긴 광역기를 완성한다.

### 4.5 반응 시간 설계 메모

숙련 타이피스트 ≈ 초당 5~7타 + 반응/판단 ≈ 0.5초. 따라서 비행 시간 1.5초인 공격에 대응할 수 있는 어구는 사실상 **3~5글자**가 한계. 결론: **반응형 방어 명령은 짧아야 한다.** 긴 정식 초식명은 선제용, 즉각 방어는 단축어 전용 — 이로써 "방어는 단축어, 비기는 풀네임"이라는 입력 문법이 자연스럽게 생긴다.

## 5. 공간 및 이동 모델

### 5.1 연속 공간 (격자 아님)

전장은 **연속 공간**이다. 헥스·사각 격자도 검토했으나 채택하지 않는다 — 격자는 실시간 전투가 턴제 감각으로 미끄러질 위험이 있고, 칸 좌표를 타이핑하는 부담이 손을 구결에서 빼앗는다. **실시간성이 핵심**이므로 연속 공간을 택한다.

### 5.2 두 레이어 — 추상(지도) vs 체화(장면)

이동조차 타이핑인 게임 특성상(이동 모델 — §5.4) 무한 맵 자유 보행은 성립하지 않는다 — 손이 구결에 묶인 채 월드를 부드럽게 돌아다닐 수 없다. 따라서 두 레이어로 분리한다.

| 레이어 | 실시간? | 내용 |
|--------|:------:|------|
| 추상(지도·메뉴) | 아니오 | 지역 노드 이동, 마을·문파·상점, 정적 대화 |
| 체화(장면) | 예 | Place에 몸을 들임 — 담판·대치·전투 등 모드를 가짐(§5.5–5.6) |

무한 맵은 채택하지 않는다. 이동은 노드 단위로 처리하고, 실시간 긴박감은 장면(체화) 레이어에 집중한다.

### 5.3 전투 아레나 — 플레이어 앵커 + 방사형 위협

- 플레이어는 대략 **중앙에 앵커**되고, 적과 장풍이 **사방에서 접근**한다.
- 접근 거리가 곧 비행 시간 = 반응 윈도우 = 칠 수 있는 구결 길이(§4.5, §9).
- "단일 위협인가 포위인가"가 한눈에 읽혀야 한다(§12 가독성).

### 5.4 이동 = 방향 + 이동법(경공)

정밀 좌표 조작은 타이핑과 양립하지 않는다. 따라서 플레이어는 **방향과 이동법(경공)만 지정**하고, 캐릭터가 알아서 이동한다. 예: *동쪽으로 제운종* → 그 방향으로 경공 특성(거리·속도)만큼 이탈.

- 이동은 매 순간의 조작이 아니라 **드물게 쓰는 전술 행위**(포위 이탈·간격 조절)다. 평소엔 방어·반격 구결에 손을 쓴다.
- 이동법(경공)마다 거리·속도·내력 소모가 다르며, 그 숙련도가 이동 성능을 좌우한다(§8 실력 축).

### 5.5 아레나 = 장소(Place) + 장면(Scene)

아레나는 추상적 전투 상자가 아니라 **강호의 실재하는 장소**(소림사 경내 입구, 취선루 2층 등)이며, 거기로 "확대"된다는 것은 칼을 빼는 게 아니라 **그 장소에 몸을 들이는 것**(체화)이다. **전투는 장면의 한 모드일 뿐이다**(§5.6).

> **장소의 정체성이 곧 전투 규칙을 낳는다.** 모든 메커닉 수치는 "이곳이 어떤 곳인가"에서 **파생**되어야지, 임의로 정해지면 안 된다.

장소가 서사적 의미를 가지면 자연히 재사용된다(1막의 비무지가 3막의 결투지로). 그래서 두 층으로 분리한다.

| 층 | 내용 | 재사용 |
|---|---|---|
| **장소(Place)** | 정체성·서사 + 그것이 낳는 물리(형태·크기·장애물·지형·출입구) | 한 번 정의, 여러 번 |
| **장면(Scene)** | 그 위에 얹는 시나리오(모드·인물·웨이브·목표·전이) | 장소마다 여럿 |

#### 좌표 — 정적은 직교, 위협 판정은 극좌표(런타임 파생)

원점은 **고정된 아레나 중심**(§5.3). 두 프레임을 구분한다:

- **배치(저장) = 직교좌표 `(x, y)`**, 중심 기준 px. 방이 사각형이므로 장애물·출구·지형·초기 위치는 직교가 정직하다. (규약: +x=우, +y=앞. 캔버스 렌더 시 y 반전)
- **위협 판정(런타임) = 극좌표, 플레이어 현재 위치 기준.** 매 프레임 `(위협 − 나)`로 계산하며 **저장하지 않는다**.

| 런타임 극좌표 축 | 의미 |
|---|---|
| 각도 θ | 방향 — "단일이냐 포위냐" 가독성(§5.3) |
| 거리 r | 비행 시간 = 반응 윈도우 = 칠 수 있는 구결 길이(§4.5, §9) |

플레이어가 중앙에 있을 땐 두 프레임이 겹치지만, **이동하면 갈라진다** — 그래서 위협 극좌표는 파생값이다.

#### Place 스키마

```jsonc
{
  "place": {
    "id": "chwiseonru_2f",
    "name": "취선루 2층",
    "region": "낙양",
    "archetype": "주루",        // 사원 | 주루 | 객잔 | 연무장 | 산길 | 동굴 | 광장 …
    "mood": "잠입·난전",
    "description": "낙양 제일 주루의 이층. 탁자가 빼곡하고 손님이 많아 운신이 좁다."
  },
  "stage": {                     // 위 정체성이 낳는 물리 — 재사용되는 무대
    "shape": "rect",
    "size": { "approachTime": 1.2 },        // 좁음 → 코앞 난전
    "obstacles": [ /* 탁자(파괴가능), 기둥 */ ],
    "terrain":   [ { "id": "cramped", "effects": { "aoeRange": 0.4 } } ],
    "exits":     [ { "pos": { "x": 0, "y": -340 }, "type": "계단" }, { "pos": { "x": 340, "y": 0 }, "type": "창문" } ]
  }
}
```

`archetype`(장소 유형)이 물리 기본값을 묶는다 — "주루"라고 정하면 좁음·탁자·손님이 프리셋으로 따라오고, 거기서 변주만 한다.

#### Scene 스키마

장면은 `mode`로 시작하고, 트리거에 따라 모드가 **흐른다**(§5.6). 전투용 `waves`는 `mode: 전투`일 때의 하위 구성이다.

```jsonc
{
  "scene": {
    "place": "chwiseonru_2f",
    "mode": "대치",                       // 담판 | 대치 | 전투
    "objective": "표적 제거, 손님 휘말림 최소화",
    "transitions": [                       // 모드는 흐른다
      { "trigger": "정체 발각", "to": "전투" },
      { "trigger": "설득 성공", "to": "담판" }
    ],
    "actors": [
      { "id": "boss", "faction": "enemy", "role": "boss",
        "spawn": { "x": 0, "y": 580, "atTime": 0 },
        "ai": "behaviorTree:geom_master", "techniques": ["장풍"], "hp": 200 },
      { "id": "guest", "faction": "neutral", "role": "protect",
        "spawn": { "x": 300, "y": 0, "atTime": 0 }, "mobile": false }
    ],
    "waves": [   // mode: 전투 격발 시
      { "atTime": 5.0, "spawn": "장풍병",   "from": { "x": 396, "y": -396 } },
      { "atTime": 8.0, "spawn": "장풍병x2", "from": [ { "x": -260, "y": -300 }, { "x": 260, "y": -300 } ] }
    ]
  }
}
```

#### 구성 4블록

1. **크기·형태** — 크기는 픽셀이 아니라 `approachTime`(외곽→중심 비행 시간) = **최대 경고 시간**. 작을수록 어렵다. `shape`: circle(방사형 최적) / fan(벽·절벽 등져 각도 제한) / rect(통로) / polygon.
2. **장애물** — 세 독립 차단 속성: `projectile`(장풍 차단→안전 각도) / `movement`(이탈 봉쇄) / `sight`(텔레그래프 차단 — **가독성 파괴, 신중히**). + `destructible`.
3. **인물** — `faction`(enemy/ally/neutral) × `role` × 극좌표 `spawn`(+`atTime` 초기/웨이브). neutral = 보호 대상(인질)이거나 방해물(행인).
4. **지형** — 핵심 레버를 구역별로 곱셈 보정: 이동 거리·속도·제어(빙판·늪) / 시야(안개·어둠) / `aoeRange`(협소·수중) / 내력(영맥·사기).

#### 장소 → 메커닉 파생 (예시)

**취선루 2층 — 잠입·난전**

| 설정(픽션) | → 메커닉 |
|---|---|
| 좁은 실내 | `approachTime 1.2초` → 긴 구결 불가, **단축어 전용 전투** |
| 탁자 빼곡 | 장애물 多(projectile+movement), 일부 파괴 가능 |
| 못 펴는 운신 | `aoeRange ×0.4` → 광역기 사실상 봉인, 정밀 견제전 |
| 손님 가득 | neutral 다수 — 휘말리면 평판/실패 |
| 창문·계단 | 특정 각도만 이탈로(exits) |
| 잠입 분위기 | 큰 소리(광역기 등) → 증원 웨이브 트리거 |

**소림사 경내 입구 — 정파·의식**

| 설정(픽션) | → 메커닉 |
|---|---|
| 넓은 마당 | `approachTime 2.5초` → 정식 초식명 시전, 정정당당한 비무 |
| 석상·향로·돌계단 | 엄폐물(일부 sight 차단) |
| 정파 성지 | 승려 ally/neutral — **중립 살해 = 큰 페널티**(살수 금기) |
| 신성한 기운 | 내력 회복(`qiRegen ×2`) |
| 의식의 무게 | 보스 비무 = **풀네임 비기 권장**(§8.4) |

#### 효과 — 장소가 무공 문법을 바꾼다

§4.5의 "방어는 단축어, 비기는 풀네임" 입력 문법이 **장소마다 강제로 달라진다**: 좁은 주루는 단축어 전용, 너른 마당은 풀네임 비기의 무대. 장소가 "어떤 입력 스타일이 요구되는가"를 결정하는 레버가 되어 전투 단조로움을 해소한다(§10.3).

#### 장소 상태 — base → variant → override

같은 장소도 시점에 따라 상태가 다르다(성업하는 취선루 vs 폐허가 된 취선루). 인물의 *클래스→override*와 똑같이 **Place도 참조 후 덮어쓴다**. 층위: **base(stage) → variant(명명된 상태) → scene placeOverride(일회성)**.

- **손님 유무·적 배치는 Scene의 actors 소관**(무인 = 안 넣으면 끝). variant/override는 장소의 *물리*(구조·지형·조명·분위기·approachTime)만 다룬다.
- obstacles·terrain은 배열이므로 **id 기준 패치**(`remove`/`modify`/`add`), 그 외 객체 필드는 deep-merge.
- **상태가 메커닉을 바꾼다:** 폐허가 되어 벽이 무너지면 `cramped`(협소) 해제 → 광역기 봉인이 풀리고 `approachTime`↑ — 같은 장소가 전혀 다른 전장이 된다.
- 시나리오의 **시간 정보**(연호·계절·시각·날씨)도 상태를 거든다 — '밤'이면 placeOverride로 시야를 낮추는 식.

### 5.6 장면의 모드 — 사교에서 살수까지

장소에 몸을 들였다고 곧 전투는 아니다. 장면은 사교에서 살수까지의 스펙트럼을 가지며, 모드 사이를 **매끄럽게 흐른다**.

| 모드 | 실시간 긴장 | 타이핑의 역할 |
|------|:---:|------|
| 담판/대화 | 낮음 | 발언·태도 선택, 기세 표출 |
| 대치(standoff) | 높음 | 아무도 아직 시전 안 함 — **첫 공격 구결이 곧 개전** |
| 전투 | 최고 | 풀 실시간(§4) |

**대치 = 타이핑 게임만의 기싸움.** 선공이란 곧 누가 먼저 공격 구결을 치느냐다. 키보드 위에 손을 얹은 채 망설이는 순간이 곧 기싸움 — 먼저 치면 선수를 잡지만 빈틈을 내고, 기다리면 상대의 텔레그래프(§9 긴 구결)를 읽는다. 전투의 수싸움이 **칼을 빼기 전부터** 작동한다.

**매끄러운 전이가 시그니처.** 별도의 "전투 시작" 화면이 없다:

```
찻잔을 나눈다(담판) → 말이 틀어진다(대치) → 누군가 첫 구결을 친다(전투)
```

실시간이기에 가능한, 로딩 없는 무협의 그 장면. 같은 Place 안에서 모드만 미끄러진다.

**비전투 장면도 공간이 의미를 갖는다.** 대치·담판·인질극·적진 호위 — 칼을 안 빼도 "누가 어디 서 있나, 출구는 어디, 누가 문을 막나"가 곧 긴장의 재료다(§5.3, §5.5 출입구·장애물).

## 6. 적 AI

적은 타이핑하지 않는다. 타이핑은 플레이어만의 인터페이스이고, 적은 정해진 타이밍에 행동하는 **룰 기반 행위자**다. 멀티플레이 없이 AI로 충분하며, 전투의 긴장감은 적의 지능이 아니라 **공격의 타이밍과 속도**에서 나온다.

### 6.1 상태 머신

```
Idle → Approach → WindUp → Attack → Cooldown → …
                       ↘ Retreat (피격 시)
                       ↘ Defend  (막혔을 때)
```

개별 적은 단순해도 된다. "똑똑해 보임"은 개체 지능이 아니라 **패턴의 조합**에서 나온다.

### 6.2 난이도 레버

AI를 더 영리하게 만들기보다 **패턴을 복잡하게** 하는 쪽으로 난이도를 조절한다.

| 레버 | 쉬움 | 어려움 |
|------|------|--------|
| 장풍 속도 | 느림(입력 여유) | 빠름(단축어 필수) |
| 동시 공격 수 | 1 | 다방향 |
| 이동·포위 패턴 | 직선 | 플레이어 포위 |
| WindUp 시간 | 길다 | 짧다 |

### 6.3 확장

- **행동 트리:** 조건 분기가 늘면 상태 머신의 상위 호환으로 전환.
- **웨이브 스크립트:** "3초 후 좌측 장풍, 5초 후 포위" 같은 연출 시퀀스(§10 캠페인 데이터).
- **반응형 패턴:** 플레이어가 특정 무공을 쓰면 대응 패턴으로 전환 — §4.4 파해와 연계.

## 7. 무공 구조 — 기본행위와 초식

### 7.1 기본행위 라이브러리 (어휘)

게임의 최소 단위. 각 행위의 **사거리·위력·속도는 그 행위의 숙련도에 따라** 달라진다.

- 이동, 회피, 구르기, 숙이기
- 발차기, 찌르기, 베기
- 장풍
- (확장)

### 7.2 초식 = 명명된 조합 (문장)

초식은 기본행위들을 엮은 **명명된 시퀀스**다. 본질적으로 함수/매크로에 가깝다. 예: `제운종 신출귀몰` = 약 10여 개 기본행위의 묶음.

- 짧은 초식명을 입력 → 캐릭터가 그 행위 시퀀스를 **실시간으로 수행**.
- **입력은 압축되지만 실행은 압축되지 않는다.** 초식의 시전 시간 = 구성 기본행위들의 수행 시간 합. → §9 밸런싱의 핵심 레버가 이 구조에서 자연히 도출된다.
- 초식의 실제 위력 = (구성 논리) × (내 기본행위들의 숙련도).

### 7.3 즉흥 vs 초식

기본행위는 항상 직접 입력 가능하게 열어둔다.

- **즉흥(기본행위 직접 조합):** 느리지만 자유롭다. 맞는 초식이 없는 상황에서 임기응변.
- **초식(학습된 조합):** 빠르지만 고정돼 있다.

초보는 `굴러` `피하기`를 따로 치다가, `굴러피하기`라는 2행위 초식을 배워 빨라지고, 점차 더 긴 초식을 손에 넣는다. 이 성장이 곧 "방어가 제때 들어가느냐(생존)"와 동일한 축이다 — 성장·난이도·메커닉·판타지가 모두 한 방향을 가리킨다.

## 8. 성장 시스템

### 8.1 성장 척추

```
획득(전수/수련) → 숙련(단련으로 mastery↑) → 단축(약어 단계 해금) → 창안 → 전수
```

무공을 얻고, 갈고닦고, 손에 익으면 줄여 쓰고, 통달하면 새로 짓고, 끝내 남에게 물려준다 — 무협 일대기 그 자체.

### 8.2 두 개의 성장 축: 지식 vs 실력

- **기본행위 = 실력(단련).** 숙련 시 사거리·위력·속도 상승. 게임의 주된 grind 대상.
- **초식 = 지식(무학).** "알고 모르고"의 문제. 새 레시피를 익히면 내 단련된 행위들이 그대로 실린다.

이 분리는 무협 정서와 정확히 겹친다: *무학 지식은 방대하나 내공이 쇠한 노고수* vs *익힌 초식은 적으나 펄펄 끓는 내공의 후기지수.*

### 8.3 획득 게이트 (수련 / 전수)

무공명을 안다고 시전할 수 있는 게 아니다. 반드시 학습 절차를 거쳐야 한다.

- **전수:** 사부·비급으로 배움 → 사회적·탐험적 루프.
- **수련:** 반복·깨달음으로 익힘 → 성장 루프.
- **미습득 무공 시전 시도 → 주화입마(자기 내력 손상).** "어설프게 아는 무공은 위험하다." (위키 보고 아무 무공명 치는 꼼수 차단 + 세계관 몰입)

### 8.4 단축 / 약어 진화

mastery가 오를수록 입력을 줄일 수 있다: `풀네임 → 약어 → 초성 → (제한적) 단일키`.

- **단축은 바닥을 낮추는 게 아니라 천장을 올린다.** 옛 적을 시시하게 만드는 게 아니라, 더 빠른 적·더 긴 콤보·더 복잡한 포위를 감당하게 해준다.
- **위력 비례 하한선** 아래로는 줄어들지 않는다(§9).
- **최상위 비기는 끝까지 풀네임을 요구.** 절정의 순간에 긴 구결을 또박또박 외게 해 의식(ritual)의 무게를 남긴다.

### 8.5 무공 창안

여러 기본행위/초식을 엮어 **나만의 명명된 초식**을 정의한다(자기 함수 정의).

규칙:
- **합성 상한선:** 한 초식에 묶을 수 있는 행위 수에 상한. 한 판 전체를 삼키는 콤보가 되지 않게, 전투 중 여러 번 명령할 이유를 남긴다(자주 쓰는 2~3초식 묶음 정도).
- **위력 비례 최소 이름 길이:** 포함 행위 수에 비례한 최소 길이. 10행위 초식을 `ㄱ` 한 글자로 만들 수 없다.
- **입력은 압축, 실행은 불변:** 짧은 내 문구로 쳐도 시전 시간·끊기 위험은 구성 행위 합만큼 그대로다. → 창안은 "콤보를 더듬는 실수를 없애는 숙련의 보상"이지 공짜 압축 치트가 아니다.
- **네임스페이스/충돌 검사:** 창안 문구가 내장 약어·기본행위 명과 충돌하지 않도록 시스템 단계에서 처리.

> **(후기 확장, 신중히)** 창안 무공을 전수 가능하게 하면 플레이어가 사부가 되고 문파를 세우며 유저 제작 무공이 메타에 퍼지는 emergent 콘텐츠가 생긴다. 잠재력은 크지만 밸런싱·악용 리스크가 커, 코어가 단단해진 뒤에 연다.

## 9. 밸런싱 프레임워크 (요약)

세 시스템이 하나의 닫힌 틀로 수렴한다.

> **위력에 비례한 (입력 길이 · 내력 · 시전 시간) 하한선 → 그 위에서 mastery가 모든 비용을 깎음 → 창안은 합성 상한 안에서 입력을 묶음.**

핵심 원칙:

- **위력 비례 하한선.** 하한선은 전역 상수가 아니라 무공의 위력(≈구성 행위 수)에 비례한다. 약한 견제기는 1~2타까지 줄어도, 사방을 쓰는 광역기는 통달해도 최소 길이가 길다. → "길이 ≈ 위력"이 모든 숙련 단계에 걸쳐 유지된다.
- **고수는 싸고 쉽게.** mastery는 입력 길이·내력·시전 시간을 모두 깎는다 — 단, 각각의 위력 비례 하한선까지만.
- **시전 시간이 최후의 제동 장치.** 길이·내력이 줄어들면 최상위 무공의 마지막 밸런스는 시전 시간(과 끊기 가능성)이 진다. 강한 무공은 통달해도 발현까지 한 박자를 남기며, 그 박자가 상대의 반격 창(窓)이다.
- **길이 = 비무 중의 정보.** 상대가 긴 구결을 또박또박 치기 시작하면 "큰 게 온다"는 텔레그래프. 끊을지·피할지·더 큰 걸로 받을지의 수싸움이 생긴다. 어구 길이가 밸런싱 화폐이자 정보가 되는 것 — 이 게임의 시그니처.

## 10. 스토리 · 캠페인 구조

### 10.1 원칙 — 습득이 곧 서사

> 새 초식을 배운다 = 새 타이핑 어휘를 얻는다 = 스토리가 진행된다.

스토리는 장식이 아니라 **무공 습득의 동기**여야 한다. 성장 척추(획득→숙련→단축→창안→전수, §8.1)가 그대로 서사의 뼈대가 된다.

### 10.2 무림 편력 3막

1. **입문** — 사부에게 기본행위를 하나씩 전수받는다. 튜토리얼이 곧 서사(`굴러`+`피하기` → `굴러피하기` 습득이 곧 승급 비트).
2. **강호** — 오픈된 탐험·수련 루프. 지역·문파마다 고유 무공 계열. 비급·도전·깨달음으로 어휘를 모은다. 지역마다 적 패턴도 달라져 단조로움을 해소.
3. **일가** — 모은 무공으로 창안하고, 끝내 제자에게 전수(§8.5, §11 후기 확장).

### 10.3 단조로움 해소 장치

| 장치 | 역할 |
|------|------|
| 비무(比武) | 보스전. 긴 구결 = 텔레그래프, 수싸움 |
| 파해 퍼즐 | 특정 적은 특정 무공으로만 깨짐 — 어휘 활용 강제 |
| 비급 해독 | 전투 외 콘텐츠. 깨달음 = 단축어 해금 |
| 정파/사파 분기 | 노선에 따라 배우는 무공 계열이 갈림 |
| 장소 재방문 | 같은 장소를 다른 장면으로 다시 — 우정의 땅이 결투의 땅으로(§5.5) |
| 대치·담판 | 칼 없이도 실시간 장면 — 첫 구결 전의 기싸움(§5.6) |

### 10.4 약점 = 동기

"지식 vs 실력" 분리(§8.2)를 서사 엔진으로 쓴다. *강한 적을 못 깬다 → 새 무공을 찾아 떠난다.* 막힘(전투 실패) → 탐험(스토리) → 습득(성장) → 돌파(전투)가 하나의 루프로 돈다.

### 10.5 캠페인 = 데이터, 엔진 = 코드

엔진(파서·전투·무공 실행·판정)은 고정 코드, **캠페인은 데이터 파일 하나**로 정의한다. 등장 무공·웨이브 스크립트·스토리 비트·해금 조건을 데이터로 기술한다.

- 기본 캠페인조차 이 포맷으로 제작 → **"캠페인 추가 = 데이터 추가"**, 코드는 그대로.
- 이 데이터 포맷이 곧 **모딩 API** → 유저 제작 캠페인/무공의 emergent 콘텐츠로 연결(§8.5 창안 전수 비전).
- **주의:** 첫 프로토타입은 하드코딩으로 핵심 루프를 검증한 뒤 데이터 포맷으로 추출한다. 처음부터 과설계하지 않되 "언젠가 데이터가 된다"는 전제로 코딩한다.

## 11. 멀티플레이 방향

실시간성이 핵심이라는 전제에서 형태별 우선순위가 갈린다.

| 형태 | 난이도 | 적합도 | 판단 |
|------|:---:|:---:|------|
| 협동(Co-op) vs AI | 중 | ★★★ | **1순위** |
| 실시간 1:1 대전 | 상 | ★ | 포기 (지연 = 불공정) |
| 비동기 PvP(리플레이) | 하 | ★★ | **보류** (실시간성과 상충) |

### 11.1 1순위 — 실시간 협동 vs AI

§2 기둥 3(직렬 입력의 한계)을 서사로 푼다: 손은 한 번에 한 문장 → **두 사람이면 두 문장**. 동료가 사각을 막고, 한쪽이 적을 묶는 사이 다른 쪽이 긴 광역기를 완성한다(§4.4 콤보).

- 적이 룰 기반 AI라 **원격 입력 지연에 관대**하다.
- 구현: 한쪽이 호스트(시뮬레이션 담당), 상대는 타이핑 명령만 전송(WebRTC P2P 또는 WebSocket 릴레이).
- 코어가 단단해진 뒤 착수한다.

### 11.2 포기 / 보류

- **실시간 1:1 대전:** 0.5초 반응 + 타이핑 속도 = 생존인 게임에서 네트워크 지연이 곧 불공정 → 처음부터 포기.
- **비동기 PvP:** 리플레이 도전은 네트워크 코드가 거의 필요 없으나, 실시간성이 핵심이라는 방향과 상충 → 보류.

## 12. 기술 설계 고려사항

- **한글 입력 처리(최우선 리스크).** 조합형·IME(한영 전환) 특성상 실시간 정오 판정이 까다롭다. 자모 단위로 볼지 완성 글자 단위로 볼지부터 결정. `compositionend` 이벤트로 글자 완성 시점을 잡아 판정하는 방식의 **조기 검증**을 강력 권장.
- **입력 파서.** 기본행위(짧은 토큰)와 초식(엔터로 확정되는 완성어)을 문법적으로 분리. 창안 문구는 네임스페이스/longest-match로 충돌 해소.
- **전장 가독성(telegraphing).** "지금이 단일 위협인지 포위인지"가 한눈에 읽혀야 한다. 상황 판단이 게임의 절반인데 그 재료가 안 읽히면 운빨이 된다.

## 13. 미해결 결정 사항 (Open Decisions)

1. **정확도 판정 시점** — 완성 후(commit, 깔끔/명확) vs 스트리밍(긴장감↑, 한글 판정 난이도↑).
2. **시전 중 플레이어 행위** — ① 지켜보기(fire-and-forget) ② 끊기/전환 허용 ③ 다음 입력 버퍼링으로 매끄러운 연계(콤보 캔슬). 전투의 질감을 좌우하는 가장 큰 결정.
3. **하한선·시전 시간의 구체 수치** — 머리로는 안 나옴. 프로토타입 플레이테스트 필요.
4. **타깃 유저** — 타이핑 애호가 vs 무협 팬. 이에 따라 난이도 게이트(타이핑 속도 벽)의 강도와 보조 장치(난이도 모드·둔화 보조)의 비중이 갈린다.
